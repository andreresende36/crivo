## 1. Contexto

O monorepo está estável: 162 testes passando, `tsc --noEmit` zero erros, 48 migrations aplicadas, Docker + entry points (`crivo-scraper`, `crivo-sender`, `crivo-api`) funcionando. O pipeline ML está em produção e o admin tem auth + gestão básica de ofertas.

A auditoria desta sessão expôs três áreas de débito acumulado e oportunidades de feature:

- `alert_bot.py` e `health_check.py` estão esqueléticos (placeholders).
- Não há tracking de clique em afiliados (perda de telemetria de conversão).
- SQLite fallback existe mas tem cobertura mínima de testes de integração.

## 2. Princípios de priorização

1. **Risco antes de feature.** Validação ausente e arquivos gigantes primeiro — base estável habilita velocidade depois.
2. **Cada item entrega algo verificável.** Critério de aceite explícito + quality gate deve passar.
3. **Máximo 1 refactor pesado por sprint.** Reduz o blast radius e mantém testes governáveis.
4. **Features pequenas vencem grandes.** Tudo abaixo é fatiável em 0.5–2 dias; nada precisa de mais que isso para ir pra `main`.

## 5. Definition of Done por feature

Cada item da sprint só é considerado *done* se cumprir:

1. **Código** — implementado seguindo as regras do CLAUDE.md (camadas Router→Service→Repository, ≤40L por função, nada de `requests` sync, `httpx` async sempre).
2. **Testes** — pelo menos 1 teste cobrindo o caminho feliz e 1 cobrindo erro/edge. Cobertura local ≥ 80% nas linhas alteradas.
3. **Quality gates** — `ruff check`, `mypy src/`, `pytest -x`, `tsc --noEmit` (admin), `next build` (admin) verdes.
4. **Migration (se aplicável)** — testada com `supabase db push` + downgrade manual (-1).
5. **Documentação** — ADR em `docs/decisions/YYYY-MM-DD-tema.md` se houve decisão arquitetural; entrada em `memory/journal/YYYY-MM-DD.md`.
6. **Sem hardcoded secrets** — toda var nova entra em `.env.example`.

## 6. Detalhamento dos itens-chave

### BE-01 · Schema Pydantic para `GET /offers`

Hoje os query params são lidos como kwargs e validados no corpo da função. Risco: tipos errados viram 500 em vez de 422.

**Como fazer:**

- Criar `packages/backend/crivo/api/schemas/offers.py` com `OffersListQuery(BaseModel)` (status, category, min_score, page, per_page).
- No router, aceitar `query: OffersListQuery = Depends()`.
- Mover lógica para `services/offers_service.py` se ainda estiver no router.

**Aceite:** request com `min_score=abc` retorna 422 com mensagem clara; teste novo cobre o caso.

### BE-05 · Extrair `ProductParser` de `ml_scraper.py`

`ml_scraper.py` mistura navegação Playwright, parsing HTML, classificação de categoria e logging. Quebra o teto de 400L e dificulta testar parsing isolado.

**Como fazer:**

- Criar `packages/backend/crivo/scraper/ml_parser.py` contendo `ProductParser` com métodos `parse_card`, `parse_price`, `parse_discount`, `parse_seller`.
- `MLScraper` passa a chamar `ProductParser`; nada novo de I/O em parser (regra do score_engine se aplica aqui — parser puro).
- Mover testes correspondentes para `tests/scraper/test_ml_parser.py`.

**Aceite:** `ml_scraper.py < 400L`; `ml_parser.py` 100% testável sem Playwright; suite continua verde.

### BE-07 · `alert_bot.py` real

Hoje é um stub. Sem ele, falhas do scraper morrem em log e ninguém sabe.

**Escopo mínimo:**

- Singleton `AlertBot` em `monitoring/alert_bot.py` recebe `TELEGRAM_ADMIN_CHAT_ID`.
- Hooks: cookies ML expirados, scraper sem novas ofertas em > 2h, sender com falha em > 3 envios consecutivos.
- Dedupe interno (mesmo alerta não dispara 2x em < 30min).

**Aceite:** simulação de cookies expirados → mensagem chega no chat de admin; teste com `httpx.MockTransport`.

### BE-08 · Tracking de cliques em afiliados

**Migration:** `049_create_click_events.sql`

```sql
create table public.click_events (
  id bigint generated always as identity primary key,
  offer_id uuid references public.offers(id) on delete cascade,
  channel text not null check (channel in ('telegram','whatsapp','web')),
  user_agent text,
  ip_hash text,           -- sha256 do IP (privacidade)
  clicked_at timestamptz not null default now()
);
create index on public.click_events (offer_id, clicked_at desc);
create index on public.click_events (channel, clicked_at desc);
```

**Endpoint:** `POST /tracking/click` recebe `{offer_id, channel, ua}`; serviço faz hash do IP e grava. Retorna 204.

**Aceite:** clique do bot Telegram passa pelo redirect → endpoint → registra evento; query `select count(*) ...` cresce.

### FE-01 · `offers/page.tsx` como Server Component

Hoje a página inteira é client; faz fetch após hydration. Migrar move o fetch para o servidor (latência menor, SEO melhor, menos JS).

**Como fazer:**

- Mover fetch Supabase para o componente RSC (usando `@supabase/ssr` server client).
- Componentes interativos (`offers-filters`, `offers-table`) ficam client; recebem dados via props.
- Empurrar `"use client"` para folhas (regra do CLAUDE.md admin).
- Adicionar `loading.tsx` na rota.

**Aceite:** mesma UI, network tab mostra HTML com dados; `next build` reporta menos JS na rota.

### FE-03 · Dashboard com KPIs reais

`(dashboard)/page.tsx` (448L) renderiza placeholders. Trocar por queries reais em Supabase.

**KPIs alvo:**

- Ofertas coletadas (24h / 7d)
- Taxa de aprovação (approved / total enviadas pra revisão)
- Top 5 categorias por volume
- CTR por canal (depende de BE-08 — entrega parcial se BE-08 não estiver pronto)

**Aceite:** todos os KPIs vêm de RPC ou view Supabase; nenhum hardcoded.
